Legacy Spec Agent

쓸 만한 문서가 전혀 없는 코드에서 근거 있는 명세를 복원하고, 이후 코드가 바뀔 때 어긋남(drift)을 감지해 그 명세를 정직하게 유지한다.

전제가 의도된 것이다 — 현실의 레거시에서 명세는 없지만 코드는 언제나 있다. 그래서 이 스킬은 명세를 입력으로 요구하지 않고 소스에서 캐낸다. 그리고 내놓는 모든 주장은 file:line 인용으로 뒷받침돼야 한다. 코드로 근거를 댈 수 없는 것은 unverified로 격리되며, 사실인 양 제시되지 않는다.

원칙 — 근거 없는 문장은 실리지 않는다. 인용 없는 규칙은 추측이고, 명세 속 추측은 빈칸보다 나쁘다. "Fact First" 규율을 그대로 옮긴 것으로, 근거가 모든 줄을 통제한다.

두 가지 모드

  • Mode A — 역생성 (기본, 레포 첫 실행): SPEC.md, ARCHITECTURE.md, audit_log.jsonl을 처음부터 만든다.
  • Mode B — Drift 추적 (이 스킬이 만든 SPEC.md가 이미 있을 때): 각 주장의 인용을 현재 코드에 다시 대조하고 DRIFT_REPORT.md를 내며 audit_log.jsonl에 덧붙인다.

워크플로 (Mode A)

Phase 0 — 스코프 & 인제스트

  • Glob으로 트리를 훑어 주 언어와 최상위 모듈/패키지를 식별한다.
  • 코드베이스가 크거나 다중 언어면 몰래 샘플링하지 말고 사용자에게 범위를 확인한다. 커버리지를 조용히 줄이지 않는다 — 범위를 좁혔으면 무엇을 뺐는지 밝힌다.
  • 모듈 목록을 만든다. 이 목록이 Phase 1의 fan-out을 이끈다.

Phase 1 — 추출 (fan-out)

모듈마다 실제로 하는 일을 뽑는다: 엔트리포인트, 비즈니스 규칙, 입출력, 부수효과, 외부 호출, 제약. 모든 항목은 path:line 인용을 단다.

  • 모듈이 여럿이면 Task 툴로 모듈당 general-purpose 서브에이전트를 하나씩 띄워 병렬 실행한다.
  • 작은 레포는 Read/Grep으로 인라인 추출한다.

Phase 2 — 아키텍트

모듈별 결과를 시스템 구조로 종합한다: 모듈 경계, 호출/데이터 흐름 엣지, 외부 의존성. ARCHITECTURE.md에 Mermaid 다이어그램을 낸다.

Phase 3 — Critic 게이트 (필수)

  • 인용된 path:line을 연다. 코드가 주장을 뒷받침하면 → verified.
  • 인용이 없거나 틀렸거나 뒷받침되지 않으면 → Unverified / Needs-review 섹션으로 옮긴다. 조용히 삭제하지 않고, 본문으로 승격하지 않는다.
  • 각 verified/flagged 결정을 audit_log.jsonl 항목으로 기록한다.

이 게이트가 "LLM에게 레포 요약시키기"와 이 스킬을 가른다. 건너뛰지 않는다.

Phase 4 — 산출

세 산출물을 쓴다. 한 문단으로 요약 보고한다: 모듈 수, verified 주장 수, unverified 수, 그리고 상위 3개 리스크/미지수.

하드 룰

  • 인용 없으면 안 실린다. 독자가 열어 확인할 수 있는 path:line 없이는 어떤 주장도 본문 명세에 넣지 않는다.
  • 격리하되 지어내지 않는다. 근거 못 대는 추론은 Unverified로 — 본문에 넣지 않고, 조용히 삭제하지도 않는다.
  • Drift 모드에서 자동 수정 금지. diff만 제안하고, 병합은 사람이 소유한다.
  • 건너뛴 것은 밝힌다. 범위를 줄였으면 커버리지 줄에 적고 숨기지 않는다.

시스템 목적

hookify는 사용자가 작성한 룰 파일을, 대화/툴 페이로드를 패턴 매칭해 툴 호출·프롬프트·정지 이벤트를 경고하거나 차단하는 Claude Code 훅으로 바꾼다. 훅 엔트리포인트가 stdin에서 이벤트 페이로드를 읽고, 해당 이벤트에 맞는 룰을 로드해 평가한 뒤, JSON 결정을 출력한다 — 동작을 절대 죽이지 않으면서. hooks/pretooluse.py:39 config_loader.py:198 rule_engine.py:35

모듈

Rule Engine — core/rule_engine.py

  • 책임: Rule 객체를 훅의 input_data에 대해 평가하고 이벤트에 맞는 응답(block / warn / allow)을 반환. rule_engine.py:35
  • 비즈니스 규칙:
    • 매칭된 룰은 rule.action == 'block'일 때만 blocking이고, 그 외 action 값은 warning으로 취급. rule_engine.py:55
    • 우선순위 — blocking 우선: blocking 룰이 하나라도 매칭되면 blocking 분기만 실행되고 warning은 버려진다. rule_engine.py:61
    • 이벤트별 block 응답: Stop → {decision:'block', reason, systemMessage}; PreToolUse/PostToolUse → permissionDecision:'deny' + systemMessage; 그 외 → {systemMessage}만. rule_engine.py:66
    • 조건이 없는 룰은 절대 매칭되지 않는다; 조건은 AND 결합. rule_engine.py:117
    • Tool matcher: '*'는 모든 툴; 아니면 |로 나눠 tool_name이 정확히 그 멤버여야 함(OR, 정규식/부분일치 아님). rule_engine.py:137
    • 지원 연산자: regex_match, contains, equals, not_contains, starts_with, ends_with; 미지원 → 매칭 안 됨. rule_engine.py:166
    • 필드 추출은 툴 인지형: Bash→command; Write/Edit→content(new_string로 폴백); MultiEdit→ 각 edit의 new_string을 공백으로 연결. rule_engine.py:231
  • 부수효과: transcript 필드는 input_data['transcript_path']를 읽음; 파일 오류는 잡혀 stderr에 경고되고 '' 반환. rule_engine.py:207
  • 제약: 모든 정규식 매칭은 대소문자 무시 — compile_regex가 re.IGNORECASE를 하드코딩(메모이즈, 최대 128). 비정규식 연산자는 대소문자 구분. rule_engine.py:14 rule_engine.py:169

Config Loader — core/config_loader.py

  • 책임: .claude/hookify.*.local.md 룰 파일(YAML frontmatter + 마크다운 본문)을 발견·읽기·파싱해 Condition/Rule 데이터클래스로 만들고, 이벤트·활성 상태로 필터. config_loader.py:198
  • 비즈니스 규칙:
    • 신형 conditions 리스트를 우선; 레거시 pattern은 conditions가 없을 때만 단일 regex_match 조건으로 변환되며 필드는 이벤트에서 추론. config_loader.py:57
    • load_rules(event)는 룰 파일을 glob하고, rule.event == 'all' 또는 == event인 활성 룰을 반환. config_loader.py:221
    • Frontmatter는 자체 라인 기반 파서(PyYAML 아님)로 파싱: 따옴표 제거; true/false를 bool로 변환; 인라인 list-dict 항목은 :와 ,를 모두 요구. config_loader.py:98
  • 제약: 탐색은 CWD 상대 — 프로세스 작업 디렉토리의 .claude 밖 룰은 안 잡힘. load_rule_file은 파일당 Rule 하나 반환. config_loader.py:210 config_loader.py:250

Hook Entrypoints — hooks/{pretooluse,posttooluse,stop,userpromptsubmit}.py

  • 비즈니스 규칙:
    • 이벤트 해석이 훅마다 다름: pretooluse/posttooluse는 Bash→bash, Edit|Write|MultiEdit→file, 그 외는 없음. hooks/pretooluse.py:46
    • stop은 고정 이벤트 'stop', userpromptsubmit은 고정 이벤트 'prompt'를 로드. hooks/stop.py:37 hooks/userpromptsubmit.py:37
    • Fail-open 보장: 예외 발생 시 Hookify 에러 메시지를 출력하고 언제나 sys.exit(0) — 훅 오류가 동작을 막지 않음. hooks/pretooluse.py:63
  • 제약: hookify 패키지 해석을 위해 CLAUDE_PLUGIN_ROOT로 sys.path 주입 필요; import 실패 시 에러 출력 후 exit 0. hooks/pretooluse.py:14

제약 & 가정 (시스템 전역)

  • 룰은 프로세스 CWD의 .claude/ 아래에만 존재; 다른 디렉토리에서 실행하면 룰이 조용히 하나도 안 잡힘. config_loader.py:210
  • 정규식은 항상 대소문자 무시; 부분일치/동등 연산자는 아님. rule_engine.py:14
  • 훅은 계약상 fail-open — 설정 오류는 세션을 깨뜨리는 대신 강제력을 약화시킴. hooks/pretooluse.py:70

⚠️ Unverified / Needs-review

코드로 완전히 근거를 댈 수 없는 항목. 사실로 취급하지 말 것.
  • action:'block'의 성숙도가 모순적. config_loader.py:40은 action을 "warn" or "block" (future)로 문서화했지만, rule_engine.py:55-61은 이미 blocking 분기를 완전히 구현. 코드는 오늘 blocking을 지원함 — 의도된 안정성 수준이 모호. (역생성이 드러낸 잠재 불일치 — 유지보수자 판단 필요)
  • 자체 YAML 파서의 엣지 입력 정확성. 이 파서는 :/,를 포함한 따옴표 값, 숫자 타입, 더 깊은 중첩을 다루지 않음; 그런 파일에서의 동작은 정확하다고 확인되지 않음. config_loader.py:98
  • posttooluse와 pretooluse는 docstring만 빼면 기능적으로 동일; 의도된 것인지 코드에 명시 없음. hooks/posttooluse.py
  • stop/prompt 룰이 실제로 에이전트를 멈출 수 있는지는 이 패키지 밖의 Claude Code 훅 계약에 달림. hooks/stop.py

벤치마크 — legacy-spec-agent vs baseline (iteration-1)

질문: 스킬이 실제로 출력을 바꾸는가, 아니면 순수 LLM도 똑같이 하는가?

방법: 같은 프롬프트, 두 갈래 — with-skill(Mode A 준수) vs baseline(동일 프롬프트, 스킬 없음). anthropics/claude-code @15a21e1의 미문서화 타겟 2개, hookify 데모와 별개. Eval B는 TypeScript로 크로스 언어 일반성 테스트.

샘플 정확도: 6/6 일치. 샘플에서 환각 라인번호는 발견되지 않음.

해석

  • 스킬의 가치는 "구조"가 아니라 근거성이다. Baseline A도 읽기 좋은 섹션 스펙을 냈지만 검증 가능한 인용이 0개였고 모르는 것을 격리하지 않았다. 잘 읽히지만 확인이 불가능하다. with-skill은 주장의 ~86–87%를 리뷰어가 열어볼 수 있는 라인에 묶고, 근거 못 대는 것은 격리한다.
  • 크로스 언어 성립. TypeScript(Eval B)도 Python과 같게 동작: 동일 커버리지, 동일 정확도, 올바른 크로스 파일 근거(import한 issue-lifecycle.ts 설정 소스를 인용).
  • 비용은 적당. with-skill은 토큰 ~14–28%와 시간을 더 쓴다 — Critic 게이트에서 라인을 다시 여는 값이다. 그 오버헤드가 곧 검증이다.
  • baseline의 실패 양상이 더 위험하다: 자신 있고 유창하며 반증 불가능한 문서. 안전한 리팩토링에 쓸 명세라면, 확인 못 하는 주장은 편의가 아니라 부채다.

이 벤치마크의 정직한 한계

  • 타겟 n=2, 각 단일 파일; fan-out 경로는 여기서 부하 테스트 안 함(hookify 데모가 커버).
  • 커버리지는 불릿에 path:line이 있으면 인용으로 셈 — 중요도 가중은 없음.
  • 정확도는 6개 인용 샘플이지 전수 감사가 아님.
  • 1 iteration, 단일 모델. 테스트셋을 넓혀 규모 있게 재실행하는 게 다음 엄밀화 단계.
각 탭은 이 저장소에 커밋된 실제 파일을 옮긴 것이다. 서술은 한글로 번역했고, 수치·인용·코드·다이어그램은 원문 그대로이므로 모두 소스에서 추적할 수 있다.
